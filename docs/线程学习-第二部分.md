
## 1. 共享数据的问题

多个线程只读取同一份数据，一般不会破坏数据状态。

真正危险的是：

```text
多个线程访问同一份数据，并且至少有一个线程会修改它。
```

复杂数据结构通常有自己的不变量。例如双向链表中，如果节点 A 的 `next` 指向节点 B，那么节点 B 的 `prev` 应该指回节点 A。

删除一个节点时，需要修改前后两个节点的指针：

```text
1. 找到要删除的节点 N
2. 修改前一个节点的 next
3. 修改后一个节点的 prev
4. 删除 N
```

在第 2 步完成、第 3 步还没完成时，链表短暂处于不一致状态。其他线程如果在这个时间点访问链表，就可能读到被破坏的不变量。

这就是共享数据在并发修改时最核心的问题。

## 2. 条件竞争

条件竞争的意思是：

```text
程序结果依赖多个线程的执行先后顺序。
```

有些竞争是良性的。例如两个线程都向队列添加任务，只要队列内部保持正确，谁先谁后都可以接受。

恶性条件竞争会破坏数据结构或程序逻辑。例如：

```text
线程 A 正在删除链表节点
线程 B 同时遍历链表
线程 B 看到中间状态
```

C++ 标准还定义了数据竞争：

```text
多个线程并发访问同一个对象，至少一个线程写，并且没有同步。
```

数据竞争会导致未定义行为。

## 3. 使用 std::mutex 保护共享数据

`std::mutex` 是 C++ 中最基本的互斥量。

不推荐手动写：

```cpp
m.lock();
// ...
m.unlock();
```

因为异常或多个返回路径可能导致 `unlock()` 被跳过。

推荐使用 RAII：

```cpp
std::lock_guard<std::mutex> guard(m);
```

`std::lock_guard` 构造时加锁，析构时自动解锁。

示例：

```cpp
#include <algorithm>
#include <list>
#include <mutex>

std::list<int> some_list;
std::mutex some_mutex;

void add_to_list(int new_value)
{
    std::lock_guard<std::mutex> guard(some_mutex);
    some_list.push_back(new_value);
}

bool list_contains(int value_to_find)
{
    std::lock_guard<std::mutex> guard(some_mutex);

    return std::find(
        some_list.begin(),
        some_list.end(),
        value_to_find
    ) != some_list.end();
}
```

关键点：

```text
std::mutex 通常作为成员变量，和被保护的数据放在一起。
std::lock_guard 写在每个访问共享数据的函数或代码块里。
只读函数也可能需要加锁，因为其他线程可能正在写。
```

## 4. 保护共享数据不能泄露引用或指针

互斥量保护的是访问路径，不是数据本身。

如果把受保护数据的指针或引用传出去，外部代码就可以绕过互斥量直接访问数据。

危险示例：

```cpp
#include <mutex>
#include <string>

class some_data
{
    int a;
    std::string b;

public:
    void do_something();
};

class data_wrapper
{
private:
    some_data data;
    std::mutex m;

public:
    template<typename Function>
    void process_data(Function func)
    {
        std::lock_guard<std::mutex> l(m);
        func(data);
    }
};

some_data* unprotected;

void malicious_function(some_data& protected_data)
{
    unprotected = &protected_data;
}

data_wrapper x;

void foo()
{
    x.process_data(malicious_function);
    unprotected->do_something();
}
```

`process_data()` 内部虽然加了锁，但 `malicious_function()` 保存了 `data` 的地址。

函数返回后锁已经释放，外部仍然能通过 `unprotected` 访问内部数据。

安全规则：

```text
不要返回受保护数据的指针。
不要返回受保护数据的引用。
不要把受保护数据引用传给不受控制的外部函数。
```

按值返回通常更安全：

```cpp
some_data get_data()
{
    std::lock_guard<std::mutex> l(m);
    return data;
}
```

这里返回的是副本，不是内部对象本身。

## 5. 接口间的条件竞争

即使每个成员函数内部都加了锁，接口设计不合理，仍然可能有条件竞争。

典型问题是 `std::stack` 风格接口：

```cpp
if (!s.empty())
{
    int value = s.top();
    s.pop();
    do_something(value);
}
```

单线程中没问题。多线程中有问题，因为 `empty()`、`top()`、`pop()` 是三个独立调用。

可能发生：

```text
线程 A: empty() 看到非空
线程 B: pop() 删除最后一个元素
线程 A: top() 访问空栈，出错
```

问题不是单个函数没有加锁，而是：

```text
检查是否为空、读取栈顶、弹出栈顶没有作为一个完整操作被保护。
```

解决方式是重新设计接口，把组合操作合并成一个操作。

## 6. 线程安全栈示例

```cpp
#include <exception>
#include <memory>
#include <mutex>
#include <stack>

struct empty_stack : std::exception
{
    const char* what() const throw()
    {
        return "empty stack!";
    }
};

template<typename T>
class threadsafe_stack
{
private:
    std::stack<T> data;
    mutable std::mutex m;

public:
    threadsafe_stack()
        : data(std::stack<T>())
    {}

    threadsafe_stack(const threadsafe_stack& other)
    {
        std::lock_guard<std::mutex> lock(other.m);
        data = other.data;
    }

    threadsafe_stack& operator=(const threadsafe_stack&) = delete;

    void push(T new_value)
    {
        std::lock_guard<std::mutex> lock(m);
        data.push(new_value);
    }

    std::shared_ptr<T> pop()
    {
        std::lock_guard<std::mutex> lock(m);

        if (data.empty())
        {
            throw empty_stack();
        }

        std::shared_ptr<T> const res(
            std::make_shared<T>(data.top())
        );

        data.pop();
        return res;
    }

    void pop(T& value)
    {
        std::lock_guard<std::mutex> lock(m);

        if (data.empty())
        {
            throw empty_stack();
        }

        value = data.top();
        data.pop();
    }

    bool empty() const
    {
        std::lock_guard<std::mutex> lock(m);
        return data.empty();
    }
};
```

使用方式：

```cpp
threadsafe_stack<int> s;

s.push(10);
s.push(20);

auto value = s.pop();

// value 是 std::shared_ptr<int>
int x = *value;
```

这里外部调用的是：

```cpp
threadsafe_stack<int>::pop()
```

而函数内部的：

```cpp
data.pop();
```

调用的是成员变量 `data` 的函数：

```cpp
std::stack<int>::pop()
```

流程：

```text
外部调用 s.pop()
进入 threadsafe_stack<T>::pop()
加锁
检查 data 是否为空
用 data.top() 复制栈顶元素
调用 data.pop() 删除内部 std::stack 的栈顶元素
返回复制出来的值
解锁
```

这个接口安全的原因是：

```text
检查、取值、删除在同一把锁里完成。
外部不能单独调用 top() 和 pop()。
不会泄露内部 data 的引用或指针。
```

## 7. 死锁

死锁的典型场景：

```text
线程 A 持有 m1，等待 m2
线程 B 持有 m2，等待 m1
```

代码类似：

```cpp
// 线程 A
m1.lock();
m2.lock();

// 线程 B
m2.lock();
m1.lock();
```

两个线程都在等待对方释放锁，程序无法继续。

当一个操作必须同时获取多把锁时，不要手动一个个加锁。

C++17 推荐使用：

```cpp
std::scoped_lock lock(m1, m2);
```

它会用避免死锁的方式一次性获取多个互斥量，并在离开作用域时自动释放。

示例：

```cpp
#include <mutex>

class some_big_object;
void swap(some_big_object& lhs, some_big_object& rhs);

class X
{
private:
    some_big_object some_detail;
    std::mutex m;

public:
    X(some_big_object const& sd)
        : some_detail(sd)
    {}

    friend void swap(X& lhs, X& rhs)
    {
        if (&lhs == &rhs)
        {
            return;
        }

        std::scoped_lock guard(lhs.m, rhs.m);
        swap(lhs.some_detail, rhs.some_detail);
    }
};
```

如果使用 C++11 或 C++14：

```cpp
std::lock(lhs.m, rhs.m);
std::lock_guard<std::mutex> lock_a(lhs.m, std::adopt_lock);
std::lock_guard<std::mutex> lock_b(rhs.m, std::adopt_lock);
```

`std::adopt_lock` 的意思是：

```text
互斥量已经被锁住，现在交给 lock_guard 管理。
```

## 8. 避免死锁的规则

常见规则：

```text
尽量避免嵌套锁。
需要多把锁时，优先使用 std::scoped_lock。
不要在持锁时调用外部未知代码。
如果必须按顺序加锁，所有线程都遵守同一顺序。
不要在持锁时等待另一个线程结束。
```

死锁不只发生在互斥量上。两个线程互相 `join()` 也会死锁：

```text
线程 A 等线程 B 结束
线程 B 等线程 A 结束
```

本质问题是形成了等待环。

## 9. 层级锁

层级锁的思想是：

```text
给每把锁一个层级值。
线程只能从高层级锁到低层级锁。
不能从低层级锁再去拿高层级锁。
```

例如：

```cpp
hierarchical_mutex high_level_mutex(10000);
hierarchical_mutex low_level_mutex(5000);
```

允许：

```cpp
std::lock_guard<hierarchical_mutex> a(high_level_mutex);
std::lock_guard<hierarchical_mutex> b(low_level_mutex);
```

不允许：

```cpp
std::lock_guard<hierarchical_mutex> b(low_level_mutex);
std::lock_guard<hierarchical_mutex> a(high_level_mutex);
```

简化实现：

```cpp
#include <climits>
#include <mutex>
#include <stdexcept>

class hierarchical_mutex
{
    std::mutex internal_mutex;

    unsigned long const hierarchy_value;
    unsigned long previous_hierarchy_value;

    static thread_local unsigned long this_thread_hierarchy_value;

    void check_for_hierarchy_violation()
    {
        if (this_thread_hierarchy_value <= hierarchy_value)
        {
            throw std::logic_error("mutex hierarchy violated");
        }
    }

    void update_hierarchy_value()
    {
        previous_hierarchy_value = this_thread_hierarchy_value;
        this_thread_hierarchy_value = hierarchy_value;
    }

public:
    explicit hierarchical_mutex(unsigned long value)
        : hierarchy_value(value),
          previous_hierarchy_value(0)
    {}

    void lock()
    {
        check_for_hierarchy_violation();
        internal_mutex.lock();
        update_hierarchy_value();
    }

    void unlock()
    {
        if (this_thread_hierarchy_value != hierarchy_value)
        {
            throw std::logic_error("mutex hierarchy violated");
        }

        this_thread_hierarchy_value = previous_hierarchy_value;
        internal_mutex.unlock();
    }

    bool try_lock()
    {
        check_for_hierarchy_violation();

        if (!internal_mutex.try_lock())
        {
            return false;
        }

        update_hierarchy_value();
        return true;
    }
};

thread_local unsigned long
    hierarchical_mutex::this_thread_hierarchy_value(ULONG_MAX);
```

`thread_local` 表示每个线程都有独立的一份 `this_thread_hierarchy_value`。

它记录当前线程已经持有的最低层级。后续只能继续获取更低层级的锁。

## 10. std::unique_lock

`std::lock_guard` 很简单：

```text
构造时加锁。
析构时解锁。
不能移动。
不能手动提前解锁。
```

`std::unique_lock` 更灵活：

```text
可以延迟加锁。
可以提前解锁。
可以重新加锁。
可以尝试加锁。
可以移动锁所有权。
```

延迟加锁示例：

```cpp
std::unique_lock<std::mutex> lock_a(lhs.m, std::defer_lock);
std::unique_lock<std::mutex> lock_b(rhs.m, std::defer_lock);

std::lock(lock_a, lock_b);
```

`std::defer_lock` 的意思是：

```text
先创建 unique_lock 对象，但暂时不加锁。
```

提前解锁示例：

```cpp
std::unique_lock<std::mutex> lock(m);

update_shared_data();

lock.unlock();

do_expensive_work();
```

如果 `do_expensive_work()` 不需要访问共享数据，就不应该继续占着锁。

## 11. 不同作用域中传递锁

`std::unique_lock` 可以移动，所以可以从函数返回。

```cpp
std::unique_lock<std::mutex> get_lock()
{
    extern std::mutex some_mutex;

    std::unique_lock<std::mutex> lk(some_mutex);
    prepare_data();

    return lk;
}

void process_data()
{
    std::unique_lock<std::mutex> lk(get_lock());
    do_something();
}
```

这个移动通常不是把锁从一个线程交给另一个线程，而是：

```text
同一个线程中，把锁的管理权从一个函数作用域移动到调用者作用域。
```

流程：

```text
process_data() 调用 get_lock()
get_lock() 加锁并准备数据
get_lock() 返回 unique_lock
process_data() 继续持有这把锁
do_something() 在锁保护下运行
process_data() 结束时自动解锁
```

`std::unique_lock` 可移动但不可复制：

```cpp
std::unique_lock<std::mutex> a(m);
std::unique_lock<std::mutex> b(std::move(a));
```

移动后 `b` 拥有锁，`a` 不再拥有锁。

## 12. 锁的粒度

锁的粒度描述一把锁保护的数据范围。

```text
粗粒度锁：一把锁保护很多数据。
细粒度锁：多把锁分别保护较小的数据区域。
```

粗粒度锁简单，但并发性能差。

细粒度锁性能可能更好，但代码复杂，死锁风险更高。

基本原则：

```text
锁要覆盖完整的逻辑操作。
持锁时间要尽量短。
不要持锁做耗时计算、文件 I/O、网络 I/O 或外部调用。
```

示例：

```cpp
void get_and_process_data()
{
    std::unique_lock<std::mutex> my_lock(the_mutex);

    some_class data_to_process = get_next_data_chunk();

    my_lock.unlock();

    result_type result = process(data_to_process);

    my_lock.lock();

    write_result(data_to_process, result);
}
```

这里 `process()` 不需要访问共享数据，所以先释放锁。

但锁得太短也可能改变语义。

示例：

```cpp
class Y
{
private:
    int some_detail;
    mutable std::mutex m;

    int get_detail() const
    {
        std::lock_guard<std::mutex> lock_a(m);
        return some_detail;
    }

public:
    Y(int sd)
        : some_detail(sd)
    {}

    friend bool operator==(Y const& lhs, Y const& rhs)
    {
        if (&lhs == &rhs)
        {
            return true;
        }

        int const lhs_value = lhs.get_detail();
        int const rhs_value = rhs.get_detail();

        return lhs_value == rhs_value;
    }
};
```

这里没有同时锁住两个对象，所以不会死锁，持锁时间也短。

但 `lhs_value` 和 `rhs_value` 不是同一时刻读取的。函数返回 `true` 只能说明两个快照值相等，不一定说明两个对象在同一瞬间相等。

## 13. 保护初始化过程：std::call_once

有些共享数据只需要初始化一次。

普通延迟初始化在多线程下不安全：

```cpp
std::shared_ptr<some_resource> resource_ptr;

void foo()
{
    if (!resource_ptr)
    {
        resource_ptr.reset(new some_resource);
    }

    resource_ptr->do_something();
}
```

多个线程可能同时发现 `resource_ptr` 为空，并重复初始化。

不要手写普通双重检查锁：

```cpp
void undefined_behaviour_with_double_checked_locking()
{
    if (!resource_ptr)
    {
        std::lock_guard<std::mutex> lk(resource_mutex);

        if (!resource_ptr)
        {
            resource_ptr.reset(new some_resource);
        }
    }

    resource_ptr->do_something();
}
```

第一次检查没有同步，可能和其他线程写 `resource_ptr` 形成数据竞争。

还可能出现：

```text
一个线程看到 resource_ptr 非空，
但 some_resource 对象内部状态还没有对当前线程安全可见。
```

推荐使用 `std::call_once`：

```cpp
std::shared_ptr<some_resource> resource_ptr;
std::once_flag resource_flag;

void init_resource()
{
    resource_ptr.reset(new some_resource);
}

void foo()
{
    std::call_once(resource_flag, init_resource);
    resource_ptr->do_something();
}
```

`std::call_once` 保证：

```text
初始化函数只执行一次。
其他线程会等待初始化完成。
call_once 返回后，初始化结果对当前线程可见。
```

类成员中的用法：

```cpp
class X
{
private:
    connection_info connection_details;
    connection_handle connection;
    std::once_flag connection_init_flag;

    void open_connection()
    {
        connection = connection_manager.open(connection_details);
    }

public:
    X(connection_info const& connection_details_)
        : connection_details(connection_details_)
    {}

    void send_data(data_packet const& data)
    {
        std::call_once(
            connection_init_flag,
            &X::open_connection,
            this
        );

        connection.send_data(data);
    }

    data_packet receive_data()
    {
        std::call_once(
            connection_init_flag,
            &X::open_connection,
            this
        );

        return connection.receive_data();
    }
};
```

C++11 起，函数内局部静态变量初始化也是线程安全的：

```cpp
my_class& get_my_class_instance()
{
    static my_class instance;
    return instance;
}
```

## 14. 保护不常更新的数据结构：std::shared_mutex

读多写少的数据结构适合使用读写锁。

`std::shared_mutex` 支持两种访问：

```text
共享锁：读者使用，多个线程可以同时持有。
独占锁：作者使用，同一时间只能一个线程持有。
```

规则：

```text
多个读者可以同时进入。
写者必须独占进入。
读者和写者不能同时进入。
多个写者不能同时进入。
```

DNS 缓存示例：

```cpp
#include <map>
#include <mutex>
#include <shared_mutex>
#include <string>

class dns_entry;

class dns_cache
{
    std::map<std::string, dns_entry> entries;
    mutable std::shared_mutex entry_mutex;

public:
    dns_entry find_entry(std::string const& domain) const
    {
        std::shared_lock<std::shared_mutex> lk(entry_mutex);

        std::map<std::string, dns_entry>::const_iterator const it =
            entries.find(domain);

        return (it == entries.end()) ? dns_entry() : it->second;
    }

    void update_or_add_entry(
        std::string const& domain,
        dns_entry const& dns_details
    )
    {
        std::lock_guard<std::shared_mutex> lk(entry_mutex);
        entries[domain] = dns_details;
    }
};
```

`find_entry()` 返回 `dns_entry`，不是引用：

```text
返回副本，避免把 entries 内部受保护的数据引用泄露出去。
```

参数：

```cpp
std::string const& domain
```

对应的是：

```cpp
std::map<std::string, dns_entry> entries;
//       ^^^^^^^^^^^
//       key 类型
```

也就是说，`domain` 是要查询的域名 key。

## 15. 嵌套锁：std::recursive_mutex

普通 `std::mutex` 不允许同一个线程重复加锁：

```cpp
std::mutex m;

m.lock();
m.lock(); // 错误，未定义行为
```

`std::recursive_mutex` 允许同一个线程多次加锁：

```cpp
std::recursive_mutex m;

m.lock();
m.lock();

m.unlock();
m.unlock();
```

锁几次，就必须解锁几次。

但不推荐把它作为常规设计方案。

常见问题：

```cpp
class X
{
private:
    std::mutex m;
    int data = 0;

public:
    void f()
    {
        std::lock_guard<std::mutex> lock(m);
        g();
    }

    void g()
    {
        std::lock_guard<std::mutex> lock(m);
        ++data;
    }
};
```

`f()` 加锁后调用 `g()`，`g()` 又加同一把锁，会出问题。

不要直接改成 `std::recursive_mutex` 逃避设计问题。

更好的做法是拆出不加锁的私有函数：

```cpp
class X
{
private:
    std::mutex m;
    int data = 0;

    void g_impl()
    {
        ++data;
    }

public:
    void f()
    {
        std::lock_guard<std::mutex> lock(m);
        g_impl();
    }

    void g()
    {
        std::lock_guard<std::mutex> lock(m);
        g_impl();
    }
};
```

原则：

```text
public 函数负责加锁。
private impl 函数假定调用者已经持有锁。
不要让 public 函数在对象不变量被破坏的中间状态下调用另一个 public 函数。
```

## 16. 第二部分总结

```text
共享数据的问题来自并发修改。
复杂数据结构在修改过程中可能短暂破坏不变量。
std::mutex 可以保护共享数据，但接口设计同样重要。
不要泄露受保护数据的指针或引用。
多个线程安全函数组合起来不一定线程安全。
需要把完整逻辑操作放在同一把锁保护下。
多把锁容易死锁，C++17 优先使用 std::scoped_lock。
std::unique_lock 更灵活，可以延迟加锁、提前解锁、移动所有权。
锁粒度要平衡正确性和性能。
只初始化一次的数据适合 std::call_once。
读多写少的数据适合 std::shared_mutex。
std::recursive_mutex 可以嵌套加锁，但通常说明设计需要调整。
```

第三章解决的是：

```text
如何保护共享数据。
```

下一步要学习的是：

```text
线程之间如何等待和通知。
```

例如线程安全栈为空时，不应该反复 `pop()` 捕获异常，而应该让线程睡眠等待，直到另一个线程 `push()` 数据后再唤醒它。这会引出 `std::condition_variable`。
