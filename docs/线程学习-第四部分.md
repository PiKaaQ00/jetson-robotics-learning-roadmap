## 1. 内存模型是什么

C++ 内存模型主要包含两层意思：

```text
内存布局：对象如何占用内存位置。
并发语义：多个线程如何安全读写这些内存位置。
```

判断数据竞争时，关键不是泛泛地说“共享数据”，而是要精确到：

```text
多个线程是否访问同一个内存位置。
是否至少有一个线程在写。
是否没有同步。
```

如果满足这些条件，就会产生数据竞争，导致未定义行为。

## 2. 对象和内存位置

C++ 程序中的数据由对象构成：

```cpp
int x;
double y;
std::string s;
RobotState state;
```

这些都是对象。

成员变量也是对象：

```cpp
struct RobotState
{
    double x;
    double y;
    double yaw;
};
```

这里：

```text
state 是对象。
state.x 是子对象。
state.y 是子对象。
state.yaw 是子对象。
```

普通标量成员通常各自有独立内存位置：

```cpp
struct Data
{
    int a;
    int b;
};
```

`a` 和 `b` 通常是两个不同内存位置。

但位域比较特殊：

```cpp
struct Flags
{
    unsigned int motor_ready : 1;
    unsigned int imu_ready : 1;
};
```

虽然 `motor_ready` 和 `imu_ready` 是两个成员，但它们可能共享同一个底层内存位置。

所以两个线程分别写它们，也可能产生数据竞争。

四个原则：

```text
每个变量都是对象，成员变量也是对象。
每个对象至少占一个内存位置。
普通基本类型各自有确定的内存位置。
相邻位域可能共享同一个内存位置。
```

ROS2 场景：

```cpp
struct RobotFlags
{
    unsigned int lidar_ready : 1;
    unsigned int camera_ready : 1;
};
```

如果多个线程分别写这些位域，可能访问同一个内存位置。更好的方式是：

```cpp
std::atomic<bool> lidar_ready{false};
std::atomic<bool> camera_ready{false};
```

或者用互斥量保护整个状态对象。

## 3. 对象、内存位置和并发

安全情况：

```text
多个线程访问不同内存位置。
多个线程只读同一个内存位置。
```

危险情况：

```text
多个线程访问同一个内存位置。
至少一个线程写。
没有互斥量、原子操作或其他同步。
```

示例：

```cpp
int counter = 0;

void increment()
{
    ++counter;
}
```

多个线程同时调用 `increment()`，如果没有同步，就是数据竞争。

用互斥量修复：

```cpp
std::mutex m;
int counter = 0;

void increment()
{
    std::lock_guard<std::mutex> lk(m);
    ++counter;
}
```

用原子类型修复：

```cpp
std::atomic<int> counter{0};

void increment()
{
    counter.fetch_add(1);
}
```

原子操作可以避免对该原子对象的数据竞争，但不一定适合维护复杂不变量。

ROS2 场景：

```cpp
std::atomic<int> dropped_frames{0};
```

相机线程：

```cpp
dropped_frames.fetch_add(1);
```

诊断线程：

```cpp
int count = dropped_frames.load();
```

这适合原子计数器。

复杂状态仍然更适合互斥量：

```cpp
std::mutex state_mutex;
RobotState state;

RobotState get_state()
{
    std::lock_guard<std::mutex> lk(state_mutex);
    return state;
}
```

## 4. 修改顺序

每个对象都有自己的修改顺序。

对同一个原子对象的所有写入，在逻辑上会排成一个顺序。

```cpp
std::atomic<int> x{0};
```

多个线程写：

```cpp
x.store(1);
x.store(2);
x.store(3);
```

所有线程必须承认 `x` 存在一个一致的修改顺序，例如：

```text
0 -> 1 -> 3 -> 2
```

但注意：

```text
每个对象有自己的修改顺序。
不同对象之间不一定有统一顺序。
```

这正是后面内存序复杂性的来源。

## 5. 原子操作和原子类型

原子操作是不可分割的操作。

其他线程不可能看到它只执行了一半。

```cpp
std::atomic<int> x{0};

x.store(10);
int v = x.load();
```

其他线程看到的只可能是旧值或新值，不会看到“写了一半”的中间状态。

原子类型定义在：

```cpp
#include <atomic>
```

常见类型：

```cpp
std::atomic<bool>
std::atomic<int>
std::atomic<unsigned>
std::atomic<size_t>
std::atomic<T*>
```

常见操作：

```cpp
load()
store()
exchange()
compare_exchange_weak()
compare_exchange_strong()
fetch_add()
fetch_sub()
fetch_or()
fetch_and()
```

原子类型通常不能拷贝：

```cpp
std::atomic<int> a{1};
// std::atomic<int> b = a; // 不允许
```

应显式读取和写入值：

```cpp
int v = a.load();
b.store(v);
```

## 6. std::atomic_flag

`std::atomic_flag` 是最简单的原子类型。

它只有两个状态：

```text
set
clear
```

初始化：

```cpp
std::atomic_flag f = ATOMIC_FLAG_INIT;
```

常用操作：

```cpp
test_and_set()
clear()
```

`test_and_set()` 的含义：

```text
读取旧值。
把标志设置为 true。
返回旧值。
```

可以用它实现自旋锁：

```cpp
class spinlock_mutex
{
    std::atomic_flag flag;

public:
    spinlock_mutex()
        : flag(ATOMIC_FLAG_INIT)
    {}

    void lock()
    {
        while (flag.test_and_set(std::memory_order_acquire))
        {
        }
    }

    void unlock()
    {
        flag.clear(std::memory_order_release);
    }
};
```

约定：

```text
flag == false：锁空闲。
flag == true：锁被占用。
```

谁能原子地把 `flag` 从 `false` 改成 `true`，谁就拿到锁。

注意：

```text
自旋锁等待时会占用 CPU。
只适合极短临界区。
ROS2 回调或耗时计算中通常不要用自旋锁。
```

## 7. std::atomic<bool>

`std::atomic<bool>` 比 `std::atomic_flag` 更实用。

它支持：

```cpp
load()
store()
exchange()
compare_exchange_weak()
compare_exchange_strong()
```

示例：

```cpp
std::atomic<bool> emergency_stop{false};

void safety_thread()
{
    emergency_stop.store(true);
}

void control_thread()
{
    if (emergency_stop.load())
    {
        stop_motor();
    }
}
```

`exchange()`：

```cpp
std::atomic<bool> b{true};

bool old = b.exchange(false);
```

执行后：

```text
old == true
b == false
```

`compare_exchange`：

```cpp
std::atomic<bool> started{false};

void start_once()
{
    bool expected = false;

    if (started.compare_exchange_strong(expected, true))
    {
        start_system();
    }
}
```

含义：

```text
如果 started 当前是 false，就把它改成 true，并返回 true。
否则返回 false，并把 expected 更新为 started 当前值。
```

适合“只允许一个线程成功”的场景。

ROS2 示例：

```cpp
std::atomic<bool> nav_started{false};

void try_start_navigation()
{
    bool expected = false;

    if (nav_started.compare_exchange_strong(expected, true))
    {
        start_navigation_stack();
    }
}
```

## 8. std::atomic<T*>

`std::atomic<T*>` 是原子指针。

它让指针本身的读写是原子的：

```cpp
std::atomic<Foo*> p;
```

支持：

```cpp
load()
store()
exchange()
compare_exchange_weak()
compare_exchange_strong()
fetch_add()
fetch_sub()
++ 
--
```

指针运算示例：

```cpp
class Foo {};

Foo some_array[5];
std::atomic<Foo*> p(some_array);

Foo* old = p.fetch_add(2);

assert(old == some_array);
assert(p.load() == &some_array[2]);
```

`fetch_add(2)` 的含义：

```text
返回旧指针。
把原子指针向后移动 2 个 Foo 元素。
```

注意：

```text
原子指针保护的是指针变量本身。
不自动保护指针指向的对象内容。
```

ROS2 中如果要原子替换大对象快照，更推荐使用原子 `shared_ptr` 操作或 C++20 的 `std::atomic<std::shared_ptr<T>>`。

## 9. 原子 shared_ptr 操作

`std::shared_ptr<T>` 本身不是原子类型，但标准库提供了原子非成员函数：

```cpp
std::atomic_load(&p)
std::atomic_store(&p, new_p)
```

示例：

```cpp
std::shared_ptr<Map> global_map;

void process_map()
{
    std::shared_ptr<Map> local =
        std::atomic_load(&global_map);

    if (local)
    {
        localize_with_map(*local);
    }
}

void update_map()
{
    auto new_map = std::make_shared<Map>(build_new_map());
    std::atomic_store(&global_map, new_map);
}
```

适合：

```text
一个线程更新不可变快照。
多个线程读取快照。
读线程拿到 shared_ptr 副本后，生命周期安全。
```

但注意：

```text
这不代表 Map 内部自动线程安全。
最好把 Map 当作不可变快照使用。
```

## 10. 内存序概览

C++ 原子操作支持 6 种内存序：

```cpp
std::memory_order_relaxed
std::memory_order_consume
std::memory_order_acquire
std::memory_order_release
std::memory_order_acq_rel
std::memory_order_seq_cst
```

大致分三类：

```text
顺序一致性：memory_order_seq_cst
获取-释放序：acquire / release / acq_rel / consume
自由序：memory_order_relaxed
```

默认内存序是：

```cpp
std::memory_order_seq_cst
```

实际建议：

```text
默认用 seq_cst，最容易理解。
性能敏感且同步关系明确时，再考虑 acquire/release。
relaxed 只适合不需要同步关系的统计计数等场景。
consume 不推荐使用。
```

## 11. 顺序一致性 memory_order_seq_cst

顺序一致性是最强内存序。

核心保证：

```text
所有 seq_cst 原子操作，在所有线程看来都有一个统一的全局顺序。
```

示例：

```cpp
std::atomic<bool> x, y;
std::atomic<int> z;

void write_x()
{
    x.store(true, std::memory_order_seq_cst);
}

void write_y()
{
    y.store(true, std::memory_order_seq_cst);
}

void read_x_then_y()
{
    while (!x.load(std::memory_order_seq_cst))
    {
    }

    if (y.load(std::memory_order_seq_cst))
    {
        ++z;
    }
}

void read_y_then_x()
{
    while (!y.load(std::memory_order_seq_cst))
    {
    }

    if (x.load(std::memory_order_seq_cst))
    {
        ++z;
    }
}
```

在顺序一致性下，不可能两个读线程都看到：

```text
自己等待的变量为 true。
另一个变量仍为 false。
```

否则会要求：

```text
x.store 在 y.store 之前
同时 y.store 在 x.store 之前
```

这是矛盾的。

## 12. 自由序 memory_order_relaxed

`memory_order_relaxed` 只保证：

```text
原子操作本身是原子的。
同一个原子变量有自己的修改顺序。
```

不保证：

```text
不同原子变量之间的可见顺序。
不同线程看到多个变量变化的顺序一致。
同步普通非原子数据。
```

示例：

```cpp
std::atomic<bool> x, y;
std::atomic<int> z;

void write_x_then_y()
{
    x.store(true, std::memory_order_relaxed);
    y.store(true, std::memory_order_relaxed);
}

void read_y_then_x()
{
    while (!y.load(std::memory_order_relaxed))
    {
    }

    if (x.load(std::memory_order_relaxed))
    {
        ++z;
    }
}
```

读线程可能看到：

```text
y == true
x == false
```

因为 `x` 和 `y` 是不同变量，`relaxed` 不保证它们之间的顺序。

适合 relaxed 的场景：

```cpp
std::atomic<int> dropped_frames{0};

void camera_callback()
{
    dropped_frames.fetch_add(
        1,
        std::memory_order_relaxed
    );
}
```

这里只需要计数不丢，不需要用计数器同步其他数据。

## 13. 获取-释放序 acquire-release

获取-释放序没有全局统一顺序，但可以成对建立同步。

典型模式：

```cpp
// 写线程
data = 42;
ready.store(true, std::memory_order_release);

// 读线程
while (!ready.load(std::memory_order_acquire))
{
}

assert(data == 42);
```

含义：

```text
release：发布之前写好的数据。
acquire：读到发布信号后，获取这些数据。
```

重要条件：

```text
acquire load 必须读到 release store 写入的值。
release 和 acquire 必须通过同一个原子变量配对。
```

ROS2 示例：

```cpp
SensorConfig config;
std::atomic<bool> config_ready{false};

void init_thread()
{
    config = load_sensor_config();
    config_ready.store(true, std::memory_order_release);
}

void worker_thread()
{
    while (!config_ready.load(std::memory_order_acquire))
    {
    }

    use_config(config);
}
```

## 14. memory_order_acq_rel

`memory_order_acq_rel` 表示：

```text
acquire + release
```

主要用于读-改-写操作：

```cpp
fetch_add()
exchange()
compare_exchange_strong()
compare_exchange_weak()
```

例子：

```cpp
std::atomic<int> sync{0};

void thread_1()
{
    data = 42;
    sync.store(1, std::memory_order_release);
}

void thread_2()
{
    int expected = 1;

    while (!sync.compare_exchange_strong(
        expected,
        2,
        std::memory_order_acq_rel))
    {
        expected = 1;
    }
}

void thread_3()
{
    while (sync.load(std::memory_order_acquire) < 2)
    {
    }

    assert(data == 42);
}
```

线程 2 同时做两件事：

```text
acquire：获取线程 1 发布的数据。
release：把同步关系继续发布给线程 3。
```

`store()` 不能用 `acq_rel`，因为只有写没有读。

`load()` 也不能用 `acq_rel`，因为只有读没有写。

## 15. memory_order_consume

`memory_order_consume` 只同步依赖于本次原子读取结果的数据访问。

示例：

```cpp
struct X
{
    int i;
    std::string s;
};

std::atomic<X*> p;
std::atomic<int> a;

void create_x()
{
    X* x = new X;
    x->i = 42;
    x->s = "hello";

    a.store(99, std::memory_order_relaxed);
    p.store(x, std::memory_order_release);
}

void use_x()
{
    X* x;

    while (!(x = p.load(std::memory_order_consume)))
    {
    }

    assert(x->i == 42);
    assert(x->s == "hello");

    // 不保证一定看到 99
    // assert(a.load(std::memory_order_relaxed) == 99);
}
```

`x->i` 和 `x->s` 依赖于 `x` 这个指针，所以理论上受 `consume` 保护。

`a` 不依赖于 `x`，所以不保证可见。

实际建议：

```text
不要使用 memory_order_consume。
使用 memory_order_acquire 替代。
std::kill_dependency 主要就是为 consume 设计的，实际也很少使用。
```

## 16. 释放序列

释放序列用于这种情况：

```text
一个 release store 后面跟着一串读-改-写操作。
后续 acquire 操作虽然读到的是中间 RMW 写入的值，
仍然可以同步到最开始的 release store。
```

示例：

```cpp
std::vector<int> queue_data;
std::atomic<int> count;

void populate_queue()
{
    unsigned const number_of_items = 20;

    queue_data.clear();

    for (unsigned i = 0; i < number_of_items; ++i)
    {
        queue_data.push_back(i);
    }

    count.store(number_of_items, std::memory_order_release);
}

void consume_queue_items()
{
    int item_index =
        count.fetch_sub(1, std::memory_order_acquire);

    if (item_index > 0)
    {
        process(queue_data[item_index - 1]);
    }
}
```

多个消费者都调用：

```cpp
count.fetch_sub(1, std::memory_order_acquire);
```

第一个消费者可能读到生产者 `store(20)` 的值。

第二个消费者可能读到第一个消费者 `fetch_sub()` 写回的值。

释放序列保证：

```text
这些消费者仍然能同步到生产者的 release store。
因此能看到 queue_data 中准备好的数据。
```

注意：这个例子用于说明内存模型，不是完整生产者-消费者队列实现。

## 17. 内存栅栏 atomic_thread_fence

内存栅栏：

```cpp
std::atomic_thread_fence(...)
```

它本身不读写变量，但会约束它前后的内存操作顺序。

示例：

```cpp
std::atomic<bool> x, y;
std::atomic<int> z;

void write_x_then_y()
{
    x.store(true, std::memory_order_relaxed);
    std::atomic_thread_fence(std::memory_order_release);
    y.store(true, std::memory_order_relaxed);
}

void read_y_then_x()
{
    while (!y.load(std::memory_order_relaxed))
    {
    }

    std::atomic_thread_fence(std::memory_order_acquire);

    if (x.load(std::memory_order_relaxed))
    {
        ++z;
    }
}
```

含义：

```text
release fence 约束它之前的写入不能跑到后面。
acquire fence 约束它之后的读取不能跑到前面。
```

栅栏位置很重要。

正确：

```cpp
x.store(true, relaxed);
atomic_thread_fence(release);
y.store(true, relaxed);
```

错误：

```cpp
atomic_thread_fence(release);
x.store(true, relaxed);
y.store(true, relaxed);
```

第二种不能把 `x` 和 `y` 分开排序。

实际建议：

```text
优先使用 store(release) / load(acquire)。
少手写 atomic_thread_fence。
```

## 18. 原子操作给非原子操作排序

原子操作不只是保护原子变量自己，也可以给普通变量建立顺序。

```cpp
bool x = false;
std::atomic<bool> y;

void writer()
{
    x = true;
    std::atomic_thread_fence(std::memory_order_release);
    y.store(true, std::memory_order_relaxed);
}

void reader()
{
    while (!y.load(std::memory_order_relaxed))
    {
    }

    std::atomic_thread_fence(std::memory_order_acquire);

    assert(x);
}
```

`x` 是非原子变量，但不会数据竞争，因为：

```text
x = true 先于 release fence。
reader 读到 y 后执行 acquire fence。
acquire fence 先于读取 x。
```

更常见的写法：

```cpp
x = true;
y.store(true, std::memory_order_release);
```

读线程：

```cpp
while (!y.load(std::memory_order_acquire))
{
}

assert(x);
```

## 19. 高级同步工具的内存语义

很多标准库工具已经内置同步语义，不需要手动写内存序。

常见规则：

```text
std::mutex::lock() 是 acquire。
std::mutex::unlock() 是 release。
std::thread::join() 与被 join 线程结束同步。
std::promise::set_value() 与 future::get() 同步。
std::packaged_task 执行完成与 future::get() 同步。
std::async 任务完成与 future::get() 同步。
condition_variable 本身不提供同步，真正同步由配套 mutex 提供。
```

互斥量示例：

```cpp
std::mutex m;
int data;

void writer()
{
    std::lock_guard<std::mutex> lk(m);
    data = 42;
}

void reader()
{
    std::lock_guard<std::mutex> lk(m);
    assert(data == 42);
}
```

不需要手动指定 `memory_order`，因为互斥量已经提供固定同步语义。

## 20. 第四部分总结

```text
内存位置是判断数据竞争的精确单位。
无同步地并发读写同一内存位置会导致未定义行为。
原子操作不可分割，可以避免对原子对象的数据竞争。
std::atomic_flag 保证无锁，但功能极少。
std::atomic<bool> 适合简单标志位。
compare_exchange 适合“只有一个线程成功改变状态”的场景。
std::atomic<T*> 保护的是指针本身，不保护指向对象。
顺序一致性 seq_cst 最强、默认、最容易理解。
relaxed 只保证原子性，不建立同步。
acquire-release 通过同一个原子变量成对建立同步。
acq_rel 适合读-改-写操作，既获取又释放。
consume 不推荐使用，用 acquire 替代。
释放序列让 release store 后的一串 RMW 继续携带同步关系。
atomic_thread_fence 可以约束内存操作顺序，但日常代码少用。
mutex、future、promise、thread::join 等高级工具已经内置同步语义。
```

本章最重要的实践建议：

```text
复杂共享数据优先使用 mutex。
等待和通知优先使用 condition_variable 或 future。
简单计数器和标志位可以使用 atomic。
不确定内存序时使用默认 seq_cst。
只有在性能敏感且能完整证明正确性时，才使用更弱内存序。
```
