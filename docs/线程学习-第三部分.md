## 1. 为什么需要等待事件

第三章解决的是：

```text
多个线程如何安全访问共享数据。
```

第四章解决的是：

```text
一个线程如何高效等待另一个线程完成某件事。
```

最差的等待方式是忙等待：

```cpp
while (!flag)
{
    // 一直检查
}
```

问题：

```text
浪费 CPU。
频繁抢锁。
可能影响真正负责设置 flag 的线程。
```

稍好一点的方式是轮询加睡眠：

```cpp
bool flag;
std::mutex m;

void wait_for_flag()
{
    std::unique_lock<std::mutex> lk(m);

    while (!flag)
    {
        lk.unlock();
        std::this_thread::sleep_for(std::chrono::milliseconds(100));
        lk.lock();
    }
}
```

这里睡眠前必须释放锁：

```cpp
lk.unlock();
```

否则其他线程无法拿到互斥量，也就无法修改 `flag`。

但这种方式仍然有问题：

```text
睡太短：浪费 CPU。
睡太长：响应变慢。
```

更合适的方式是：

```text
条件不满足时线程睡眠。
条件满足时由另一个线程通知它醒来。
```

这就是条件变量。

## 2. std::condition_variable

`std::condition_variable` 用来等待某个条件成立。

典型组合：

```cpp
std::mutex
std::condition_variable
std::unique_lock
```

示例：生产者线程准备数据，消费者线程等待数据。

```cpp
#include <condition_variable>
#include <mutex>
#include <queue>

std::mutex mut;
std::queue<data_chunk> data_queue;
std::condition_variable data_cond;

void data_preparation_thread()
{
    while (more_data_to_prepare())
    {
        data_chunk const data = prepare_data();

        {
            std::lock_guard<std::mutex> lk(mut);
            data_queue.push(data);
        }

        data_cond.notify_one();
    }
}

void data_processing_thread()
{
    while (true)
    {
        std::unique_lock<std::mutex> lk(mut);

        data_cond.wait(lk, [] {
            return !data_queue.empty();
        });

        data_chunk data = data_queue.front();
        data_queue.pop();

        lk.unlock();

        process(data);

        if (is_last_chunk(data))
        {
            break;
        }
    }
}
```

`wait()` 的行为可以理解为：

```text
检查条件。
如果条件不满足，释放互斥量并让线程睡眠。
被 notify_one 或 notify_all 唤醒。
重新获取互斥量。
再次检查条件。
条件满足后返回，并继续持有锁。
```

这里必须用 `std::unique_lock`，因为 `wait()` 需要在内部解锁和重新加锁。`std::lock_guard` 不够灵活。

为什么要传谓词：

```cpp
data_cond.wait(lk, [] {
    return !data_queue.empty();
});
```

原因是条件变量可能伪唤醒。线程醒来不代表条件一定成立，所以必须重新检查条件。

ROS2 场景：

```text
/camera/image_raw 回调线程：收到图像后 push 到队列。
vision_worker 后台线程：等待队列非空，取出图像做目标检测。
```

## 3. 构建线程安全队列

`std::queue` 的接口不适合直接用于多线程：

```cpp
if (!q.empty())
{
    auto value = q.front();
    q.pop();
}
```

这在多线程中有接口竞争：

```text
线程 A：empty() 看到非空。
线程 B：pop() 弹出最后一个元素。
线程 A：front() 出错。
```

所以线程安全队列要把：

```text
检查
取值
删除
等待
```

封装进一个完整操作。

常用接口：

```cpp
template<typename T>
class threadsafe_queue
{
public:
    void push(T new_value);

    bool try_pop(T& value);
    std::shared_ptr<T> try_pop();

    void wait_and_pop(T& value);
    std::shared_ptr<T> wait_and_pop();

    bool empty() const;
};
```

完整版示例：

```cpp
#include <condition_variable>
#include <memory>
#include <mutex>
#include <queue>

template<typename T>
class threadsafe_queue
{
private:
    mutable std::mutex mut;
    std::queue<T> data_queue;
    std::condition_variable data_cond;

public:
    threadsafe_queue() = default;

    threadsafe_queue(threadsafe_queue const& other)
    {
        std::lock_guard<std::mutex> lk(other.mut);
        data_queue = other.data_queue;
    }

    threadsafe_queue& operator=(threadsafe_queue const&) = delete;

    void push(T new_value)
    {
        {
            std::lock_guard<std::mutex> lk(mut);
            data_queue.push(std::move(new_value));
        }

        data_cond.notify_one();
    }

    void wait_and_pop(T& value)
    {
        std::unique_lock<std::mutex> lk(mut);

        data_cond.wait(lk, [this] {
            return !data_queue.empty();
        });

        value = std::move(data_queue.front());
        data_queue.pop();
    }

    std::shared_ptr<T> wait_and_pop()
    {
        std::unique_lock<std::mutex> lk(mut);

        data_cond.wait(lk, [this] {
            return !data_queue.empty();
        });

        std::shared_ptr<T> res(
            std::make_shared<T>(std::move(data_queue.front()))
        );

        data_queue.pop();
        return res;
    }

    bool try_pop(T& value)
    {
        std::lock_guard<std::mutex> lk(mut);

        if (data_queue.empty())
        {
            return false;
        }

        value = std::move(data_queue.front());
        data_queue.pop();
        return true;
    }

    std::shared_ptr<T> try_pop()
    {
        std::lock_guard<std::mutex> lk(mut);

        if (data_queue.empty())
        {
            return std::shared_ptr<T>();
        }

        std::shared_ptr<T> res(
            std::make_shared<T>(std::move(data_queue.front()))
        );

        data_queue.pop();
        return res;
    }

    bool empty() const
    {
        std::lock_guard<std::mutex> lk(mut);
        return data_queue.empty();
    }
};
```

`mutable std::mutex mut` 的原因：

```text
empty() 是 const 函数。
但加锁会修改 mutex 内部状态。
所以 mutex 需要 mutable。
```

ROS2 场景：

```text
订阅回调只负责快速 push 数据。
后台线程 wait_and_pop 数据并处理。
```

例如：

```cpp
threadsafe_queue<ImageFrame> image_queue;

void image_callback(ImageFrame const& image)
{
    image_queue.push(image);
}

void vision_worker()
{
    while (true)
    {
        ImageFrame image;
        image_queue.wait_and_pop(image);
        run_detector(image);
    }
}
```

## 4. std::future

`std::future<T>` 表示：

```text
将来会得到一个 T 类型的结果。
```

它可以等待异步任务完成并获取结果。

最简单的方式是 `std::async`：

```cpp
#include <future>
#include <iostream>

int calculate()
{
    return 42;
}

int main()
{
    std::future<int> result =
        std::async(std::launch::async, calculate);

    do_other_stuff();

    std::cout << result.get() << "\n";
}
```

`get()` 的行为：

```text
如果任务已完成，立即返回结果。
如果任务没完成，阻塞等待。
普通 std::future 的 get() 通常只能调用一次。
```

`std::launch::async` 表示：

```text
明确要求任务异步执行。
```

如果只写：

```cpp
std::async(calculate);
```

标准库可以选择异步执行，也可以延迟到 `get()` 或 `wait()` 时再执行。

ROS2 场景：

```cpp
std::future<Path> path_future =
    std::async(std::launch::async, [goal] {
        return plan_path(goal);
    });

update_robot_state();

Path path = path_future.get();
publish_path(path);
```

## 5. std::packaged_task

`std::packaged_task` 的作用是：

```text
把一个函数包装成任务，并把任务结果绑定到 future。
```

它不决定任务在哪个线程执行，适合任务队列和线程池。

基本例子：

```cpp
#include <future>
#include <iostream>

int calculate()
{
    return 42;
}

int main()
{
    std::packaged_task<int()> task(calculate);
    std::future<int> result = task.get_future();

    task();

    std::cout << result.get() << "\n";
}
```

任务队列例子：

```cpp
std::mutex task_mutex;
std::condition_variable task_cond;
std::deque<std::packaged_task<Path()>> planner_tasks;

void planner_thread()
{
    while (true)
    {
        std::packaged_task<Path()> task;

        {
            std::unique_lock<std::mutex> lk(task_mutex);

            task_cond.wait(lk, [] {
                return !planner_tasks.empty();
            });

            task = std::move(planner_tasks.front());
            planner_tasks.pop_front();
        }

        task();
    }
}

std::future<Path> post_planning_task(Goal goal)
{
    std::packaged_task<Path()> task([goal] {
        return plan_path(goal);
    });

    std::future<Path> result = task.get_future();

    {
        std::lock_guard<std::mutex> lk(task_mutex);
        planner_tasks.push_back(std::move(task));
    }

    task_cond.notify_one();

    return result;
}
```

关键顺序：

```text
post_planning_task() 创建任务，但不执行 plan_path。
任务被移动进队列。
planner_thread() 被唤醒。
planner_thread() 取出任务。
planner_thread() 调用 task()。
这时才真正执行 plan_path(goal)。
结果自动进入 future。
```

## 6. std::promise

`std::promise` 的作用是：

```text
一个线程手动设置结果，另一个线程通过 future 等待结果。
```

基本关系：

```cpp
std::promise<int> p;
std::future<int> f = p.get_future();
```

```text
promise：放结果的一端。
future：取结果的一端。
```

示例：

```cpp
#include <future>
#include <iostream>
#include <thread>

void worker(std::promise<int> p)
{
    p.set_value(42);
}

int main()
{
    std::promise<int> p;
    std::future<int> f = p.get_future();

    std::thread t(worker, std::move(p));

    int result = f.get();

    std::cout << result << "\n";

    t.join();
}
```

适合场景：

```text
网络响应到达。
ROS2 service 返回。
ROS2 action 完成。
硬件初始化完成。
某个回调里拿到了结果。
```

ROS2 示例：

```cpp
std::promise<bool> nav_done_promise;
std::future<bool> nav_done_future =
    nav_done_promise.get_future();

void navigation_result_callback(bool success)
{
    nav_done_promise.set_value(success);
}

void task_thread()
{
    bool ok = nav_done_future.get();

    if (ok)
    {
        start_grasp_task();
    }
}
```

## 7. 将异常存入 future

`future` 不只可以传返回值，也可以传异常。

```cpp
double square_root(double x)
{
    if (x < 0)
    {
        throw std::out_of_range("x < 0");
    }

    return std::sqrt(x);
}
```

异步调用：

```cpp
std::future<double> f =
    std::async(std::launch::async, square_root, -1);

try
{
    double y = f.get();
}
catch (std::out_of_range const& e)
{
    std::cout << e.what() << "\n";
}
```

异常发生在异步任务中，但会在：

```cpp
f.get();
```

这里重新抛出。

`std::promise` 手动传异常：

```cpp
std::promise<double> p;

try
{
    p.set_value(calculate_value());
}
catch (...)
{
    p.set_exception(std::current_exception());
}
```

已知异常类型时：

```cpp
p.set_exception(
    std::copy_exception(std::logic_error("invalid data"))
);
```

如果 `promise` 被销毁前没有 `set_value()` 或 `set_exception()`，等待方调用 `future.get()` 会得到 `std::future_error`，错误状态是 broken promise。

## 8. std::shared_future

`std::future` 适合一个线程等待一个结果。

多个线程等待同一个结果时，用：

```cpp
std::shared_future<T>
```

示例：

```cpp
std::promise<int> p;
std::shared_future<int> sf = p.get_future().share();

std::thread t1([sf] {
    int value = sf.get();
    use_value_1(value);
});

std::thread t2([sf] {
    int value = sf.get();
    use_value_2(value);
});

p.set_value(42);

t1.join();
t2.join();
```

注意：

```text
每个线程最好持有自己的 shared_future 副本。
不要多个线程同时访问同一个 shared_future 对象。
```

ROS2 场景：

```text
地图加载完成后，定位、规划、可视化多个模块都要等待同一份地图。
```

```cpp
std::promise<Map> map_promise;
auto map_future = map_promise.get_future().share();

std::thread localization([map_future] {
    Map map = map_future.get();
    start_localization(map);
});

std::thread planner([map_future] {
    Map map = map_future.get();
    start_global_planner(map);
});

map_promise.set_value(load_map());
```

## 9. std::chrono 与超时

C++ 时间工具在：

```cpp
#include <chrono>
```

常用时钟：

```text
std::chrono::system_clock：系统真实时间，可能被调整。
std::chrono::steady_clock：稳定递增，适合超时和测量间隔。
std::chrono::high_resolution_clock：高精度时钟，不一定稳定。
```

常用时间段：

```cpp
std::chrono::milliseconds(100)
std::chrono::seconds(2)
```

C++14 字面量：

```cpp
using namespace std::chrono_literals;

auto period = 10ms;
auto timeout = 500ms;
```

`future` 超时等待：

```cpp
using namespace std::chrono_literals;

auto status = path_future.wait_for(100ms);

if (status == std::future_status::ready)
{
    Path path = path_future.get();
    publish_path(path);
}
else if (status == std::future_status::timeout)
{
    publish_status("planning still running");
}
```

`wait_for()` 的含义：

```text
最多等待 100ms。
如果提前 ready，就提前返回。
不会取出结果。
真正取结果要调用 get()。
```

条件变量绝对超时：

```cpp
std::condition_variable cv;
bool done = false;
std::mutex m;

bool wait_loop()
{
    auto const timeout =
        std::chrono::steady_clock::now() +
        std::chrono::milliseconds(500);

    std::unique_lock<std::mutex> lk(m);

    while (!done)
    {
        if (cv.wait_until(lk, timeout) ==
            std::cv_status::timeout)
        {
            break;
        }
    }

    return done;
}
```

用 `wait_until()` 的原因：

```text
循环中使用 wait_for() 可能因为伪唤醒反复重置等待时长。
使用固定截止时间可以保证整体等待时间有限。
```

ROS2 场景：

```cpp
bool wait_for_odom()
{
    std::unique_lock<std::mutex> lk(odom_mutex);

    return odom_cv.wait_for(lk, 200ms, [] {
        return has_odom;
    });
}
```

如果 200ms 没有里程计，可以进入安全模式。

## 10. 函数式并发

函数式编程强调：

```text
函数结果只依赖输入参数。
不修改外部共享状态。
相同输入得到相同输出。
```

这种风格适合并发，因为：

```text
少共享可变数据。
少互斥量。
少条件竞争。
```

快速排序的并行思想：

```text
取 pivot。
分成 lower 和 higher。
lower 异步排序。
higher 当前线程排序。
最后 get lower 结果并合并。
```

示例：

```cpp
template<typename T>
std::list<T> parallel_quick_sort(std::list<T> input)
{
    if (input.empty())
    {
        return input;
    }

    std::list<T> result;
    result.splice(result.begin(), input, input.begin());

    T const& pivot = *result.begin();

    auto divide_point = std::partition(
        input.begin(),
        input.end(),
        [&](T const& t) {
            return t < pivot;
        });

    std::list<T> lower_part;
    lower_part.splice(
        lower_part.end(),
        input,
        input.begin(),
        divide_point);

    std::future<std::list<T>> new_lower =
        std::async(
            &parallel_quick_sort<T>,
            std::move(lower_part));

    auto new_higher =
        parallel_quick_sort(std::move(input));

    result.splice(result.end(), new_higher);
    result.splice(result.begin(), new_lower.get());

    return result;
}
```

核心思想：

```text
能用输入参数和返回值解决的，就不要让多个线程直接修改同一份共享数据。
```

ROS2 示例：

```cpp
auto ground_future = std::async(std::launch::async, [cloud] {
    return segment_ground(cloud);
});

auto obstacle_future = std::async(std::launch::async, [cloud] {
    return cluster_obstacles(cloud);
});

auto freespace_future = std::async(std::launch::async, [cloud] {
    return detect_free_space(cloud);
});

auto ground = ground_future.get();
auto obstacles = obstacle_future.get();
auto freespace = freespace_future.get();

publish_perception_result(ground, obstacles, freespace);
```

多个线程共享读取 `cloud`，但不共享修改同一个结果对象。

## 11. 消息传递同步

消息传递的思想：

```text
线程之间不共享可变状态。
每个线程维护自己的状态。
线程之间通过消息队列通信。
```

ATM 状态机例子：

```cpp
struct card_inserted
{
    std::string account;
};

class atm
{
    messaging::receiver incoming;
    messaging::sender bank;
    messaging::sender interface_hardware;
    void (atm::*state)();

    std::string account;
    std::string pin;

    void waiting_for_card()
    {
        interface_hardware.send(display_enter_card());

        incoming.wait()
            .handle<card_inserted>(
                [&](card_inserted const& msg)
                {
                    account = msg.account;
                    pin = "";
                    interface_hardware.send(display_enter_pin());
                    state = &atm::getting_pin;
                });
    }

    void getting_pin();

public:
    void run()
    {
        state = &atm::waiting_for_card;

        try
        {
            for (;;)
            {
                (this->*state)();
            }
        }
        catch (messaging::close_queue const&)
        {
        }
    }
};
```

状态函数只做两件事：

```text
接收消息。
根据消息更新自己的内部状态，并发送新消息。
```

ROS2 本身就很接近消息传递模型：

```text
topic
service
action
callback
executor
```

机器人任务管理示例：

```text
WAIT_FOR_OBJECT
NAVIGATING_TO_OBJECT
ALIGNING
GRASPING
DONE
ERROR
```

收到不同消息后切换状态：

```text
object_detected -> NAVIGATING_TO_OBJECT
navigation_success -> ALIGNING
alignment_done -> GRASPING
grasp_success -> DONE
navigation_failed -> ERROR
```

## 12. 持续性并发、when_all 和 when_any

并发技术扩展规范中的 `std::experimental::future` 支持：

```cpp
future.then(...)
```

含义：

```text
future ready 后，自动执行下一步。
```

适合连续流水线任务：

```text
加载地图 -> 构建 costmap -> 初始化规划器 -> 开始导航
```

注意：

```text
then() 是 experimental 扩展，不是标准 std::future 的成员函数。
```

`when_all` 的思想：

```text
多个 future 全部 ready 后，再执行汇总。
```

适合：

```text
点云分块处理后汇总。
多个传感器初始化完成后启动系统。
多个感知模型全部完成后融合结果。
```

`when_any` 的思想：

```text
多个 future 中任意一个 ready，就先处理。
```

适合：

```text
多个路径规划器并行，谁先得到可行路径就先用谁。
多个网络请求，谁先响应就使用谁。
```

## 13. latch 和 barrier

锁存器 `latch`：

```text
一次性倒计时同步对象。
计数减到 0 后永久 ready。
不能重置复用。
```

适合：

```text
等待一批初始化事件全部完成。
等待多个线程都准备好数据。
```

概念示例：

```cpp
void foo()
{
    unsigned const thread_count = ...;
    latch done(thread_count);
    my_data data[thread_count];

    std::vector<std::future<void>> threads;

    for (unsigned i = 0; i < thread_count; ++i)
    {
        threads.push_back(std::async(std::launch::async, [&, i] {
            data[i] = make_data(i);
            done.count_down();
            do_more_stuff();
        }));
    }

    done.wait();
    process_data(data, thread_count);
}
```

`done.wait()` 等的是：

```text
所有 data[i] 都准备好。
```

不是等所有线程完全结束。

栅栏 `barrier`：

```text
可复用同步点。
一组线程每一轮都必须全部到达。
最后一个线程到达后，所有线程一起继续。
```

适合分阶段并行处理：

```text
线程 0 拆分数据。
所有线程等待数据拆分完成。
每个线程处理自己的块。
所有线程等待处理完成。
线程 0 汇总输出。
进入下一轮。
```

ROS2 场景：

```text
latch：等待相机、雷达、IMU、底盘全部初始化完成。
barrier：一帧点云分成 N 块，N 个线程处理，每帧结束后统一汇总。
flex_barrier：每轮处理完成后自动执行融合结果和读取下一帧。
```

## 14. 第三部分总结

```text
condition_variable 用来等待条件成立，避免忙等待。
线程安全队列封装 mutex、queue 和 condition_variable。
future 表示将来的结果。
async 可以启动带返回值的异步任务。
packaged_task 把函数包装成任务，适合任务队列和线程池。
promise 允许一个线程手动设置结果，另一个线程通过 future 获取。
future 可以保存异常，get() 时重新抛出。
shared_future 适合多个线程等待同一个结果。
chrono 提供时间段、时间点、超时等待和周期控制。
函数式并发通过输入参数和返回值减少共享数据。
消息传递通过队列和状态机减少共享可变状态。
when_all/when_any 解决多个 future 的组合等待问题。
latch 适合一次性等待多个事件。
barrier 适合多线程分阶段反复同步。
```

第四章的核心可以概括为：

```text
第三章关注如何保护共享数据。
第四章关注线程之间如何等待、通知、传结果和组织任务。
```
